## 📌 Решённые проблемы (для справки)

- [x] **CHAT_SUBSCRIBE обязателен** — без явной подписки (opcode=75) на каждый чат push-уведомления NOTIF_MESSAGE (opcode=128) не приходят. Решено: подписка при старте поллинга.
- [x] **Вложенный payload** — данные сообщения лежат в `event["payload"]["message"]`, а не в корне payload. Решено: `msg = p.get("message", p)`.
- [x] **Persistent-соединение** — ранее создавалось новое TCP-соединение каждые 10 секунд. Решено: одно долгоживущее соединение.
- [x] **Резолвинг имён** — числовые sender ID заменяются на имена через `data/user_names.json`.
- [x] **Форматирование текста** — Rich-элементы (bold, italic, links) из Max конвертируются в HTML через UTF-16 offset/length.
- [x] **NOTIF_MESSAGE opcode** — исправлен с 110 на 128.
- [x] **Динамическая подписка на новые чаты** — бот периодически сверяет список чатов, создаёт topic для новых чатов и ставит подписку без ручного рестарта.
- [x] **Скачивание видео Max -> TG** — используется `download_video(...)`, видео отправляется в Telegram как `send_video`, а не как превью.
- [x] **Отправка видео TG -> Max** — реализованы `VIDEO_UPLOAD` (`opcode=82`), upload локального видео и отправка `VIDEO` attach в Max.
- [x] **Гонка video processing в Max** — после upload может приходить `errors.process.attachment.video.not.processed`. Решено: retry в `send_local_video(...)` до готовности вложения.
- [x] **Эхо TG -> Max -> TG** — сообщения, отправленные через bridge в Max, не должны возвращаться в Telegram вторым кругом. Решено: suppression по подтверждённому Max `message_id`.
- [x] **Хардкод имён чатов** — topic name определяется из данных чата, без прежних специальных строк в `sync()`.
- [x] **Автоматический резолвинг имён диалогов** — topic title для личных чатов строится по `login_payload["contacts"]` + `participants`, поэтому `Чат <id>` больше не должен появляться при наличии контакта.
- [x] **Кэширование `user_names.json`** — добавлен in-memory cache через `lru_cache`.
- [x] **Процессы-зомби** — локальное управление переведено на `data/bot.pid` + `scripts/start_bot.sh` / `scripts/stop_bot.sh`.
- [x] **Typed events из upstream MaxAPI** — bridge переведён на `MaxMessageEvent`, `MaxDeleteEvent`, `MaxContactEvent`.
- [x] **History API из upstream MaxAPI** — используется для controlled replay последних сообщений при создании topic и при `rebuild_topics.py`.
- [x] **Файлы/документы** — `Max FILE <-> Telegram document` реализовано в обе стороны.
- [x] **Редактирование сообщений** — `TG -> Max` и `Max -> TG` propagation реализованы, с safe replace там, где Telegram не позволяет in-place edit.
- [x] **Удаление сообщений** — `NOTIF_MSG_DELETE` обрабатывается для `Max -> TG`.
- [x] **Long-lived polling runner из upstream MaxAPI** — используется `MaxPollingRunner`, а polling изолирован в отдельный worker process.
- [x] **Удалить debug print()** — runtime-диагностика переведена на `logger`.
- [x] **Docker runtime изоляция Max SDK** — Max polling и Max SDK RPC вынесены в отдельные subprocesses (`max-polling`, `max-sdk-worker`), чтобы `SIGSEGV` в GOST/LibreSSL не убивал Telegram runtime.
